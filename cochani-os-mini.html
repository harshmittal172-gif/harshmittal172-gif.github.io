Case Study № 02

An AI operations assistant, in one Python file.

A working, shareable proof that the operator can also build. Upload messy sales data, get LLM-generated insights on a live local dashboard. Roughly a hundred lines of code, written between shifts.

Role
Sole engineer · personal project
Built
June 2026 — Two days
Stack
Python · Anthropic Claude API · Streamlit · pandas
The problemProof, not claim

Most commercial people can't show their technical work. I wanted to.

The reframing was simple: rather than describe the AI tool I'd built at Cochani, ship a small public version of the idea — one that anyone reviewing my profile could clone, run, and inspect. Code on the table, no pitch deck required.

i / iii
The buildEnd-to-end in two evenings

From a raw API to a working tool on localhost.

The application accepts a CSV upload, summarises the dataset, and submits a structured prompt to Anthropic's Claude API. The response — three specific, commercially-useful actions — renders on a Streamlit dashboard at localhost:8501. The architecture is deliberately small: the point was to walk the full path from API key to working interface, not to over-engineer.

What I learned by doing it: prompt design as a commercial skill, environment variables for key handling, and the discipline of building something a non-technical end-user could actually pick up and use.

ii / iii
The artefactPublic, runnable

Live on GitHub. Cloneable. Inspectable.

The repository ships with sample retail data, a README that explains the architecture, and a single command to run. The bar I set was: a recruiter or engineer should be able to clone it, see what it does, and read the code in ten minutes. View on GitHub →
